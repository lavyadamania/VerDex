# 🔄 REAL-TIME DATA SOURCES & INTEGRATION GUIDE

## Overview

The real-time live monitoring system displays events that are automatically generated when actions occur in the system. This guide explains where data comes from, how it flows through the system, and how to populate your database.

---

## 📊 Where Real-Time Data Comes From

### 1. **User Actions** (Primary Source)
Events generated when authenticated users interact with the system:

| User Action | Event Type | Who Triggers | Where |
|---|---|---|---|
| Change case status | `STATUS_UPDATE` | Court Staff / Admin | Case detail page → Update status button |
| Upload document | `DOCUMENT_UPLOADED` | Victim / Advocate / Staff | Case detail → Documents section |
| Schedule hearing | `HEARING_SCHEDULED` | Court Staff | Case → Add hearing date |
| Add adjournment | `ADJOURNMENT` | Court Staff | Case → Court events |
| Issue judgment | `JUDGMENT` | Judge / Court Staff | Case → Add judgment |
| Add admin note | `ADMIN_NOTE` | Admin | Case → Internal notes |

**Flow:**
```
User Action → Express Route → Model Save → emitCaseEvent() → MongoDB + Redis → Socket.io → Frontend
```

### 2. **Automated System Events** (Secondary Source)
Events generated by background jobs/workers:

| Automatic Event | Trigger | When | Frequency |
|---|---|---|---|
| `DELAY_ALERT` | Case exceeds time threshold | 180+ days pending | Hourly scan (hearingReminder worker) |
| `STAGNATION_FLAG` | No updates in long period | 12+ months inactive | Daily scan (stagnationDetector) |
| `HEARING_STARTED` | Court room opens case | Manual notification | On-demand |

**Flow:**
```
Scheduled Job → Check Database → Find Matches → emitCaseEvent() → MongoDB + Redis → Socket.io → Frontend
```

### 3. **External Integrations** (Future Enhancement)
Potential data sources:
- Court management system APIs
- E-filing portals
- Case tracking from other courts
- Lawyer association updates
- News/media case mentions

## 📈 Real-Time Event Generation

### Understanding Event Types

#### **PUBLIC EVENTS** (Visible to everyone, including non-logged-in visitors)
```javascript
rolesVisibleTo: ['admin', 'court_staff', 'advocate', 'victim', 'visitor']
```

Examples:
- Case filed
- Hearing scheduled
- Judgment issued
- Documents uploaded

**Visible in:** Public dashboard, Live feed (unauthenticated)

#### **RESTRICTED EVENTS** (Visible only to staff and involved parties)
```javascript
rolesVisibleTo: ['admin', 'court_staff', 'advocate', 'victim']
usersVisibleTo: [victim_id]  // Only specific victim
```

Examples:
- Internal status updates
- Private case notes

**Visible in:** Dashboard (login required)

#### **ADMIN-ONLY EVENTS** (Internal use only)
```javascript
rolesVisibleTo: ['admin', 'court_staff']
```

Examples:
- Admin notes
- System stagnation flags
- Internal delay alerts

**Visible in:** Admin dashboard only

## 🔴 Real-Time Socket.io Flow

### For Logged-In Users

```
1. User logs in → Gets JWT token
2. Browser connects to Socket.io with token
3. Server authenticates token
4. User joins rooms: user_<id>, role_<role>, case_<id>
5. Events published → Redis → Socket.io → User's browser
6. LiveMonitoringCard updates in real-time
```

### For Public/Visitors

```
1. User visits /dashboard/public (no login)
2. Browser connects to Socket.io without token
3. Server assigns 'visitor' role
4. Visitor joins room: role_visitor
5. Only PUBLIC events received
6. LiveMonitoringCard shows progress updates
```

## 🚀 How to Trigger Events

### Method 1: Via UI (User Actions)

1. **Status Update**
   - Go: Case detail → Edit → Change status
   - Events: `STATUS_UPDATE` emitted
   - Visible to: All assigned users

2. **Document Upload**
   - Go: Case → Documents → Upload file
   - Events: `DOCUMENT_UPLOADED` emitted
   - Visible to: All case participants

3. **Add Hearing**
   - Go: Case → Court Events → Add hearing
   - Events: `HEARING_SCHEDULED` emitted
   - Visible to: Public (if marked as public)

### Method 2: Via API (Direct)

```bash
# Update case status
curl -X PATCH http://localhost:5000/api/cases/CASE_ID/status \
  -H "Authorization: Bearer TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "status": "hearing",
    "next_hearing_date": "2025-12-31T10:00:00Z"
  }'

# Upload document
curl -X POST http://localhost:5000/api/documents/CASE_ID/upload \
  -H "Authorization: Bearer TOKEN" \
  -F "document=@file.pdf" \
  -F "doc_type=evidence"
```

Each of these automatically emits a real-time event!

### Method 3: Via Demo Data Script

```bash
npm run seed:demo
# Creates sample events instantly
```

## 📱 Frontend Integration

### Showing Real-Time Events to Non-Logged Users

#### 1. **Public Dashboard (No Login)**
```javascript
// frontend/src/pages/PublicDashboardPage.jsx
function PublicDashboardPage() {
  // Users access at: http://localhost:5173/dashboard/public
  // No authentication needed
  
  return (
    <>
      <LiveMonitoringCard />  {/* Shows public events in real-time */}
      <CaseList />            {/* Public case summaries */}
      <StaticCharts />        {/* Anonymous statistics */}
    </>
  )
}
```

#### 2. **Live Event Hook (Works for Visitors Too)**
```javascript
// frontend/src/hooks/useLiveEvents.js
export default function useLiveEvents() {
  const token = localStorage.getItem('ct_token')  // May be null for visitors
  
  const socket = io(apiBase, {
    auth: {
      token: token || '',  // Empty string for visitors
    }
  })
  
  // Socket.io server allows empty token → assigns 'visitor' role
  // Visitor only receives events with 'visitor' in rolesVisibleTo
}
```

#### 3. **API Endpoints Data Flow**
```
Initial Load:
  useLiveEvents() → GET /api/events/public → 20 latest events

Real-Time Updates:
  Socket.io → 'live_event' emission → New events in state

For Authenticated Users:
  GET /api/events/live → Role-filtered events
  GET /api/events/case/:id → Case-specific events
```
